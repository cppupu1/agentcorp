# AgentCorp - 产品现状分析与迭代演进计划

**文档目标**：基于产品设计文档（PRODUCT_DESIGN.md）及当前代码库实现的实际调研，从产品经理视角出具的一份现状诊断与后续演进落地的敏捷规划方案。

---

## 一、 当前产品开发现状核实

通过对底层代码库的调研，我们确认了目前核心业务逻辑的实现进度，这为我们制定后续的迭代计划提供了坚实的基础：

1. **异步交互与长耗时任务处理**：系统已经实现了基于 Server-Sent Events (SSE) 的状态推送（`SSEManager`），能够在前端实时展现任务进度、子任务状态和工具调用，用户无需盲等，交互底层已就绪。
2. **热干预与人工审批（Human-in-the-loop）**：底层执行器（`task-executor.ts`）已支持任务暂停（Pause）、重试（Retry）机制，并且实现了特有的 `ask_user_decision` 工具，允许 AI 在遇到歧义或需要授权时主动挂起任务等待用户输入。这初步实现了“热干预”的能力。
3. **场景模板（Templates）的成熟度**：系统中已经预置了如“软件开发”、“内容创作”等多套成熟的团队场景模板（`templates.ts`），为降低用户首发门槛提供了良好基础。
4. **HR 助手的能力边界**：目前 HR 助手（`hr-assistant.ts`）主要聚焦于以对话方式引导用户创建和配置单个“AI 员工”（提取角色设定、系统提示词等），尚未跃升为统筹建队、派发任务的全局入口。
5. **记忆系统与 RBAC 权限**：代码库中已出现 `memory.ts`，说明记忆系统的基础框架正在搭建中；而 RBAC 目前仅体现在工具调用的基础访问级别（读/写）控制上，企业级复杂权限模型暂未过度设计，这非常符合敏捷开发的节奏。

---

## 二、 核心壁垒与高价值功能（保持与强化）

产品设计在以下几个维度上展现了极强的 B 端竞争力，必须在后续开发中坚定不移地保持并强化：

1. **“拟人化”的企业隐喻**
   - 采用 PM、员工、团队的组织架构，将复杂的 Agent 编排（DAG/工作流）隐藏在“团队协作模式（流水线、辩论等）”的背后，极大地降低了业务人员的认知门槛。
2. **防范 "17x Error Trap" 的隔离与中转设计**
   - 员工间禁止直接通信，由 PM 充当信息路由器进行审查和分发。结合底层的结构化输出校验，有效截断了多 Agent 协作中最致命的“幻觉级联放大”风险。
3. **极度务实的成本与安全护栏**
   - 无论是“双模式（自动/建议）”运行，还是对 Token 与执行时间的细粒度追踪，都直击企业老板使用 AI 时的“成本失控”和“安全越界”两大核心痛点。

## 三、 关键体验风险与破局点

尽管底层已有支撑，但在迈向真实业务落地的过程中，体验层面上仍存在以下需要重点攻克的瓶颈：

### 1. 从“长尾等待”到“真正的异步委托”
虽然有 SSE 实时动画，但多节点并发与流转依然可能耗时数分钟。
- **痛点**：用户一旦关闭网页或切换网络，可能面临“任务断线”或“结果未送达”的焦虑。
- **破局点**：强化全局的消息触达（Webhook/邮件/站内信）。明确传达“任务已托管”的心智，将前端的“实时监控画布”降级为“可选项”，而非“必看项”。

### 2. “热干预”的低阻力化
底层虽然有了 `ask_user_decision`，但交互上如果不够顺滑，就会变成烦人的“弹窗拦截”。
- **痛点**：如果 AI 走偏了，用户只能通过文字回复去纠正，沟通成本高。
- **破局点**：在任务画布上支持直接双击修改某个 AI 的中间态交付物（如：修改生成的大纲），然后一键“基于此结果继续往下执行”，这比纯对话纠偏更符合业务人员的习惯。

### 3. “冷启动”的陡峭学习曲线
- **痛点**：目前 HR 助手虽然能建“员工”，但用户其实不想要“员工”，他们想要的是“帮我解决这个问题”。如果每次都要从配模型、建员工、组团队开始，流失率会极高。

## 四、 敏捷迭代建议路线图（Agile Roadmap）

基于现状，建议在寻找 PMF 的关键时期，对原有的 Phase 规划做如下微调，将有限的研发资源聚焦在“惊艳的首屏体验”和“越用越聪明的记忆”上。

### 阶段一：打透“一键发包”体验（当前核心目标）

**策略：弱化“配置”，强化“开箱即用”。**
1. **强化 MagicInput "一键发包" 链路**：
   - 目前已有的 `MagicInput` + `/api/ai/parse-intent` 链路是极其宝贵的资产。与其改造 HR 助手，不如将 MagicInput 升级为全局的超级入口。用户只需输入：“帮我写一份下半年的营销策划。” 系统自动解析意图，匹配内置的“内容创作团队模板”，自动实例化相关员工与团队，并直接抛出包含任务拆解的执行计划让用户一键点下“同意并开工”。
   - HR 助手则继续保持其清晰的边界，专注于协助用户深度定制和管理“AI 员工”。
2. **模板市场的一级化**：
   - 把预置的模板（研发、数据分析等）放到首页最显眼的位置，实现真正的“零配置试跑”。
3. **完善异步通知闭环**：
   - 即使页面关闭，任务完成后能在系统内形成一份结构化的交付物报告，并配上明显的红点通知（进一步强化现有的 `NotificationsPage`）。

### 阶段二：建立“记忆壁垒”与“柔性干预”

**策略：让平台具有复利价值，越用越好用。**
1. **记忆系统 (Memory System) 降级与MVP验证**：
   - 考虑到当前验证 PMF 的优先级最高，不要急于做重度的“完整复盘提取”。先跑通 MVP，即在任务层面增加简单的“经验标签”和关键踩坑记录，让 PM Agent 下次执行同类任务时能简单规避即可。
2. **可视化画布的热干预能力**：
   - 允许用户在“等待审批”状态下，不仅能点“通过/驳回”，还能直接对 AI 生成的 JSON/Markdown 中间件进行富文本编辑，修改后直接放行流转（目前的 `ReviewPanel` 和 `ObserverPanel` 已在这方面探索，需进一步强化该体验）。

### 阶段三：可观测性、质量保障与生态边界

**策略：满足企业级管理与治理需求。**
1. **完善 AI 质量保障体系与治理**：
   - 目前底层已经存在 `testScenarios`、`testRuns` 表以及 `policyPackages` 的基础设施，这意味着实现 AI 测试 AI 的质保体系和企业级策略包的落地成本将远低于预期。可以快速将这些底层能力在 UI 层面对用户暴露。
2. **企业级治理**：
   - 在此阶段再逐步引入复杂的 RBAC（多组织/项目权限模型），结合现有的工具级别控制，构建完整的安全边界。
3. **开放协议（A2A）与外部工具集成**：
   - 进一步扩展 MCP 支持，将团队能力以服务化（API/A2A）形式向外输出。

---
**总结**：
AgentCorp 的设计非常惊艳，底层核心基础设施（SSE, Executor, Templates）也已搭建成型。接下来最大的挑战不是技术深度，而是**包装的艺术**——如何通过强化现有的 MagicInput 解析链路和内置模板，把复杂的“多 Agent 编排引擎”包装成一个“傻瓜式的企业服务外包大厅”。一旦这个体验闭环跑通，产品的商业价值将不可估量。
