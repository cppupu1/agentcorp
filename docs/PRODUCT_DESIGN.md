# AgentCorp - AI团队协作平台 产品设计文档

## 一、产品愿景

构建一个让用户能够轻松创建和管理拟人化AI团队的平台。不同于传统的固定工作流编排，AgentCorp让AI像真实的人类团队一样协作——有项目经理统筹全局，有专业成员各司其职，能根据实际情况动态调整策略，真正实现智能化的团队协作。

## 二、产品定位与竞品差异化

### 2.1 市场格局

当前AI Agent生态可分为三层：

| 层级 | 代表产品 | 特点 | 局限 |
|------|----------|------|------|
| 开发者框架 | CrewAI、AutoGen、LangGraph | 灵活强大，支持复杂多Agent编排 | 需要编程能力，无可视化界面 |
| 低代码Agent平台 | Dify、Coze | 可视化搭建，易上手 | 偏向单Agent+固定工作流，缺乏真正的多Agent动态协作 |
| Agent可观测性 | Langfuse、AgentOps | 监控追踪Agent行为 | 只做监控，不做编排 |

### 2.2 AgentCorp的定位

AgentCorp定位为「面向业务用户的多Agent团队协作平台」，填补市场空白：

- 与开发者框架的区别：无需编程，通过可视化界面创建和管理AI团队
- 与低代码平台的区别：不是固定工作流编排，而是由PM Agent动态决策的真正团队协作
- 核心差异化：拟人化团队隐喻（员工、PM、团队），让非技术用户也能理解和使用多Agent系统

### 2.3 协议兼容性

AgentCorp拥抱开放协议标准，确保与更广泛的AI生态互通：

**MCP（Model Context Protocol，Anthropic）**：
- 定位：解决Agent→Tool的连接问题
- 当前支持：工具管理模块已采用MCP标准（npx方式启动MCP Server）
- 价值：用户可以接入MCP生态中的海量工具，无需自行开发

**A2A（Agent-to-Agent Protocol，Google）**：
- 定位：解决不同系统的Agent之间的互通问题
- 核心概念：AgentCard（能力声明）、Task（任务生命周期）、Artifact（交付物）、Message（消息）
- 规划支持：AgentCorp的"团队"可以通过A2A协议对外暴露为一个Agent服务，也可以调用外部A2A兼容的Agent
- 价值：用户的AI团队不局限于平台内部，可以与外部Agent生态互通

**协议分工**：
- MCP负责"员工能用什么工具"（向下连接工具和数据）
- A2A负责"平台与外部Agent系统的互通"（对外连接其他Agent平台）
- 注意：A2A是对外互通协议，不改变平台内部的通信规则。平台内部员工之间的通信始终通过PM中转（详见第七章任务执行部分），这是保证信息质量和错误隔离的核心设计

## 三、核心理念

**拟人化 = 大脑 + 性格 + 技能 + 记忆**

每个AI员工都是一个完整的"数字人"：
- 大脑（LLM模型）：提供思考和推理能力
- 性格与职责（System Prompt）：定义其专业领域、行为风格、关注点
- 技能（Tools）：赋予其与外部世界交互的能力
- 记忆（Knowledge & Memory）：提供专业知识和经验积累

## 四、用户角色

| 角色 | 描述 |
|------|------|
| 平台用户 | 创建和管理AI团队，发布任务，监控执行 |
| PM员工（AI） | 团队的项目经理，负责需求对齐、任务拆解、团队协调 |
| 普通员工（AI） | 团队成员，执行具体任务 |
| 观察者（AI） | 监控团队运行质量，检测异常 |

## 五、功能模块设计

### 5.1 模型管理

**目的**：管理可用的LLM模型，作为员工的"大脑"供选择。

**功能点**：
- 添加模型配置：Base URL、模型名称/标识、API Key
- 模型列表：展示所有已配置的模型，显示名称、提供商、状态
- 模型测试：配置完成后可一键测试连通性和基本能力
- 模型编辑/删除：支持修改配置和移除模型
- 兼容性：优先兼容OpenAI标准接口格式

**模型配置字段**：
- 显示名称（用户自定义，如"GPT-4o"、"Claude Sonnet"）
- Base URL（API端点地址）
- 模型标识（实际调用时的model参数）
- API Key（加密存储）
- 备注（可选）

### 5.2 工具管理

**目的**：管理AI员工可使用的外部工具，扩展其能力边界。

**功能点**：
- 添加工具：支持MCP工具（npx方式启动）的配置
- 工具配置：名称、描述、启动命令、参数配置、环境变量
- 工具测试：配置后可测试工具是否正常运行
- 工具分组：支持按类别分组管理（如"数据获取"、"文件操作"、"通信"等）
- 工具状态：显示工具的可用状态

**工具配置字段**：
- 工具名称
- 工具描述（会被LLM读取，需清晰描述工具能力）
- 启动方式（npx命令）
- 启动参数
- 环境变量（如API密钥等，加密存储）
- 所属分组（可选）

### 5.3 员工管理

**目的**：创建和管理AI员工，每个员工是一个具备独立能力的智能体。

**功能点**：

**创建员工**：
- 基本信息：员工名称、头像（可自动生成）、简介
- 大脑选择：从已配置的模型中选择一个作为该员工的LLM
- 工具分配：从已配置的工具中选择该员工可使用的工具集
- 系统提示词：定义员工的角色、职责、行为准则
  - 手动编写：用户直接编写系统提示词
  - AI生成：用户输入员工名称和角色描述，由AI自动生成专业的系统提示词
  - 混合模式：AI生成初稿，用户在此基础上修改
- 标签系统：
  - 自动标签：员工使用的模型名称自动成为标签
  - 用户标签：用户自定义标签，如"PM"、"QA"、"分析师"、"开发者"等
  - 标签用于筛选和快速识别员工角色

**员工操作**：
- 编辑：修改员工的任何配置
- 复制：一键复制员工，生成副本供用户修改（适合创建相似角色）
- 删除：移除员工（被团队引用的员工需先解除引用）
- 测试对话：可以直接与单个员工进行对话测试，验证其表现

**员工列表**：
- 卡片式展示，显示头像、名称、标签、模型、工具数量
- 支持按标签筛选、按名称搜索
- 支持列表/网格视图切换

### 5.4 员工导入导出

**目的**：支持员工配置的便携性和共享性。

**功能点**：
- 导出：
  - 支持勾选多个员工批量导出
  - 导出为JSON格式文件
  - 导出内容包含：员工基本信息、系统提示词、标签、工具配置引用
  - 不导出敏感信息（API Key等）
- 导入：
  - 支持导入JSON格式的员工配置文件
  - 一次可导入多名员工
  - 导入时检测工具依赖，提示用户配置缺失的工具
  - 导入时检测模型依赖，提示用户配置缺失的模型
  - 支持导入预览，用户确认后再执行导入
- 员工市场（远期）：
  - 社区共享的员工配置库
  - 用户可发布自己的员工配置供他人使用
  - 支持评分和评论

### 5.5 知识库管理

**目的**：为员工提供专业知识支撑，让AI不只靠通用能力，还能基于特定领域知识做判断。

**功能点**：
- 创建知识库：名称、描述、所属领域
- 知识导入：
  - 支持上传文档（PDF、Word、Markdown、TXT）
  - 支持输入网页URL抓取内容
  - 支持手动录入知识条目
- 知识库与员工绑定：一个员工可以挂载多个知识库
- 知识检索：基于向量检索，员工在思考时自动检索相关知识
- 知识库更新：支持增量更新，不影响已有内容

### 5.6 团队管理

**目的**：将员工组织成团队，定义协作结构。

**功能点**：

**创建团队**：
- 团队基本信息：名称、描述、适用场景
- 成员配置：
  - 必须设置至少一名员工
  - 必须指定一名员工为项目经理（PM）
  - 可添加其他员工作为团队成员
  - 可添加观察者角色（可选）
- 工具授权：设置团队级别的可用工具范围（团队内员工只能使用团队授权的工具）
- 协作模式选择（详见第五章）

**团队操作**：
- 编辑团队配置
- 复制团队
- 删除团队
- 查看团队历史任务

**团队列表**：
- 展示团队名称、成员数量、PM、协作模式、历史任务数
- 支持搜索和筛选

## 六、协作模式

团队的协作模式决定了成员之间如何配合完成任务。用户创建团队时选择一种基础协作模式，PM在实际执行中可以在此基础上动态调整。

### 6.1 自由协作模式（默认）

- PM完全自主决定任务拆解和分配方式
- 最灵活，适合探索性任务或难以预定义流程的场景
- 适用场景：创意类工作、研究分析、复杂问题求解

### 6.2 流水线模式

- 任务按预定义的阶段顺序流转：阶段A → 阶段B → 阶段C
- 每个阶段由指定员工负责，上一阶段的输出是下一阶段的输入
- PM负责阶段间的质量把关和衔接
- 适用场景：软件开发（需求→设计→编码→测试→部署）、内容生产（选题→撰写→审核→发布）

### 6.3 辩论模式

- 多个员工对同一问题独立分析，给出各自的判断和依据
- 分析完成后进入交叉审查阶段，员工互相质疑和补充
- PM汇总各方观点，形成最终结论
- 适用场景：投资决策、风险评估、方案选型

### 6.4 投票/共识模式

- 多个员工独立对同一问题做出判断
- 系统汇总所有判断结果，按多数原则或加权方式得出结论
- 如果分歧过大，PM介入协调或提交用户决策
- 适用场景：分类判断、合规审查、质量评估

### 6.5 主从模式

- 一个主员工负责规划和汇总，多个从员工并行执行子任务
- 从员工的结果汇总到主员工进行整合
- 适用场景：大规模数据采集与分析、批量处理任务

### 6.6 混合模式

- 在一个任务中组合使用多种协作模式
- 例如：先用流水线模式完成数据收集，再用辩论模式进行分析决策
- PM根据任务阶段动态切换协作方式

## 七、任务管理

### 7.1 任务创建流程

任务创建是一个用户与PM对话协作的过程，分为以下阶段：

**第一阶段：需求对齐**
1. 用户选择一个团队，创建新任务
2. 用户用自然语言描述任务目标
3. PM与用户进行多轮对话，澄清需求细节：
   - 任务的具体目标是什么？
   - 期望的交付物是什么形式？
   - 有哪些约束条件？
   - 优先级和质量要求？
4. PM将对齐的内容整理为结构化的「任务书」

**任务书包含**：
- 任务标题
- 任务目标（清晰、可衡量）
- 交付物定义（类型、格式、质量标准）
- 约束条件
- 验收标准

**第二阶段：用户审批任务书**
1. PM将任务书呈现给用户
2. 用户可以修改、补充或直接批准
3. 批准后进入团队组建阶段

**第三阶段：团队组建**
1. PM根据任务需求，从团队成员中推荐参与本次任务的员工
2. PM说明每个推荐员工的理由和预期职责
3. 如果现有员工不能满足需求，PM可以：
   - 建议创建临时员工（定义角色、系统提示词、分配工具）
   - 临时员工仅在本次任务中存在，任务结束后用户可选择是否保留为系统员工
4. 用户可以调整团队成员：增加、移除、替换
5. 确认团队成员后，PM为每个参与员工生成「任务上下文提示词」：
   - 基于员工原有的系统提示词
   - 追加本次任务的特定要求和上下文
   - 注意：不修改系统级的员工配置，仅在本次任务中生效

**第四阶段：任务规划**
1. PM制定任务执行计划：
   - 将任务拆解为子任务
   - 为每个子任务分配负责员工
   - 定义子任务之间的依赖关系（串行/并行）
   - 设置关键审批节点
2. 执行计划呈现给用户确认
3. 用户批准后开始执行

### 7.2 任务执行

#### 双模式运行

每个任务支持两种运行模式，可在任务创建时设置，也可由系统根据风险等级自动选择：

**建议模式（Suggest Mode）**：
- 员工只产出分析和建议，不执行任何外部操作
- 所有工具调用仅限只读类（查询、搜索、分析）
- 最终交付物是建议方案，由用户决定是否执行
- 默认适用场景：金融交易、资金操作、合规审查等高风险任务

**自动模式（Auto Mode）**：
- 员工可以执行被授权的所有操作，包括写操作
- 关键操作仍受审批节点和护栏机制约束
- 适用场景：数据采集、内容生成、代码编写等低风险任务

**自动切换逻辑**：
- 系统根据任务涉及的工具类型自动判断风险等级
- 涉及资金、外部写操作、敏感数据的任务自动切换为建议模式
- 用户可以手动覆盖系统的自动判断（需二次确认）
- PM在执行过程中也可以对特定子任务切换模式

#### 分级上线机制

从建议模式过渡到自动模式不是一步到位，而是分四个阶段渐进放开，降低风险：

**第一阶段：仿真模式（Simulation）**
- 完全模拟执行，不产生任何真实操作
- 所有工具调用被拦截，返回模拟结果
- 用于验证执行计划的合理性和员工的行为是否符合预期
- 适用场景：新团队首次执行任务、高风险任务的预演

**第二阶段：影子模式（Shadow）**
- 与真实系统并行运行，但不实际写入
- 员工产出的操作指令与真实系统的实际操作进行对比
- 用于评估AI决策与人工决策的一致性
- 适用场景：团队能力验证期、新策略包的试运行

**第三阶段：受限自动模式（Limited Auto）**
- 允许执行真实操作，但有严格的范围限制
- 例如：只允许操作测试环境、只允许小额交易、只允许发布到草稿箱
- 每次操作都记录详细日志，供事后审计
- 适用场景：团队已通过仿真和影子验证，开始小范围实操

**第四阶段：全量自动模式（Full Auto）**
- 在策略包约束下执行所有被授权的操作
- 关键操作仍保留审批节点
- 适用场景：团队已积累足够的成功执行记录，用户充分信任

**阶段晋升规则**：
- 每个阶段需要达到一定的成功执行次数和质量评分才能晋升
- 用户可以手动跳过阶段（需确认风险提示）
- 出现严重问题时自动降级到上一阶段

**执行机制**：
- PM按照执行计划调度各员工执行子任务
- 并行子任务同时启动，串行子任务按依赖顺序执行
- 每个员工执行子任务时：
  1. 接收PM分配的子任务描述和所需上下文
  2. 独立思考和执行（可调用被授权的工具）
  3. 产出子任务结果，返回给PM
- PM负责：
  - 监控各子任务的执行进度
  - 审查子任务的输出质量
  - 处理异常情况（员工执行失败、输出不符合预期等）
  - 在需要时动态调整计划（新增子任务、重新分配等）
  - 协调员工之间的信息传递

**员工间通信**：
- 员工之间不直接通信，所有信息通过PM中转
- PM负责筛选和整理信息，确保每个员工只收到与其任务相关的上下文
- 这样设计的好处：PM可以把控信息质量，避免错误信息在团队中传播

**动态调整**：
- PM在执行过程中可以根据实际情况调整计划
- 新增子任务、调整依赖关系、重新分配员工
- 所有动态调整都会在任务监控面板中实时反映

### 7.3 任务监控

**监控面板**：
提供实时的、可视化的任务执行状态展示。

**整体视图**：
- 画布式布局，展示所有子任务节点和它们之间的依赖关系
- 节点样式根据状态变化：
  - 待执行（灰色）
  - 执行中（蓝色，带动画）
  - 等待审批（黄色）
  - 已完成（绿色）
  - 失败/异常（红色）
- 依赖关系用有向连线表示
- 支持画布缩放和拖拽

**子任务节点信息**：
- 子任务名称和简要描述
- 负责员工（头像+名称）
- 当前状态
- 开始时间
- 输入来源（依赖哪些上游子任务）
- 输出摘要（完成后显示）

**员工状态面板**（侧边栏）：
- 列出所有参与员工
- 每个员工显示：当前状态（空闲/工作中/等待中）、正在执行的子任务、已完成的子任务数
- 点击员工可查看其详细的执行日志

**实时日志**：
- 展示任务执行的时间线
- 记录关键事件：子任务开始/完成、PM决策、异常处理、动态调整等
- 支持按员工筛选日志

**审批节点**：
- 当执行到需要用户审批的节点时，系统暂停并通知用户
- 用户可以查看当前成果，选择批准继续或要求修改

### 7.4 任务交付

**交付物管理**：
- 每个任务定义明确的交付物类型：
  - 文档类：分析报告、设计方案、会议纪要等
  - 数据类：数据表格、统计结果等
  - 代码类：代码文件、配置文件等
  - 决策类：决策建议（含结论、依据、风险提示、置信度）
  - 执行类：操作执行结果和日志
- 交付物有结构化格式，不只是纯文本
- PM在任务完成时汇总所有交付物，生成任务总结报告

**证据链要求**：

所有交付物中的结论和判断必须附带完整的证据链，确保可审计、可追溯：

- 依据来源：结论基于哪些数据、文档或分析结果，需标注具体来源
- 来源质量评级：每个数据源需标注可信度评级
  - A级（权威来源）：官方公告、监管文件、经审计的财务报告等
  - B级（可靠来源）：主流媒体、知名研究机构报告、行业数据库等
  - C级（参考来源）：社交媒体、论坛、未经验证的第三方数据等
  - 结论不能仅依赖C级来源，必须有至少一个A级或B级来源支撑
- 时效性标记：数据和信息的获取时间，并标注时效SLA
  - 实时数据：标注获取时间精确到分钟
  - 日频数据：标注日期
  - 低频数据：标注发布周期和下次更新时间
  - 超过时效SLA的数据需标注"可能过时"警告
- 置信度评估：对每个关键结论给出置信度（高/中/低），并说明影响置信度的因素
- 反例与风险：主动列出与结论相矛盾的证据或可能推翻结论的风险因素
- 冲突证据处理：当多个来源的证据相互矛盾时，按来源质量评级确定优先级，并明确标注冲突点
- 推理链路：从原始数据到最终结论的推理步骤，中间不能有"黑箱"

证据链格式示例（决策类交付物）：
```
结论：建议增持XX股票
置信度：中（65%）
依据：
  1. [A级来源] 2026年Q4财报，营收同比增长23% — 获取时间：2026-02-15 | 时效：季频，下次更新2026-05
  2. [B级来源] 行业研报《XX行业2026展望》— 发布时间：2026-01-20 | 时效：年频
  3. [B级来源] 近30日资金流向数据 — 获取时间：2026-02-16 | 时效：日频
反例/风险：
  1. 管理层近期有减持行为，可能暗示内部信心不足
  2. 行业监管政策存在不确定性，Q2可能出台新规
冲突证据：依据1（财报向好）与反例1（管理层减持）存在矛盾，依据1为A级来源优先采信，但反例1不可忽视
推理链路：财报数据向好 + 行业趋势正面 → 基本面支撑 → 但管理层减持 + 政策风险 → 综合置信度中等
```

**任务归档**：
- 完成的任务自动归档
- 可查看历史任务的完整记录：任务书、执行过程、交付物
- 支持任务搜索和筛选
- 支持交付物对比（如对比两次分析的结论差异）

## 八、触发器机制

### 8.1 触发方式

**手动触发**：
- 用户在界面上手动创建任务（默认方式）

**定时触发**：
- 设置Cron表达式，按时间周期自动创建任务
- 示例：每天早上9点触发运营日报生成、每周一触发周报汇总
- 支持设置触发时的任务参数模板

**Webhook触发**：
- 提供Webhook URL，外部系统通过HTTP请求触发任务
- 示例：代码仓库PR事件触发代码审查、监控系统告警触发问题分析
- 支持配置请求验证（签名校验）
- Webhook携带的数据可作为任务的输入参数

### 8.2 触发配置

- 每个触发器绑定一个团队
- 可设置任务模板：预定义任务目标和参数，触发时自动填充
- 可设置是否跳过需求对齐阶段（对于标准化的重复任务）
- 可设置触发频率限制，防止过度触发

## 九、护栏与策略包机制

护栏不再是零散的规则配置，而是封装为「策略包」——针对不同场景预设的一整套安全和合规规则，用户一键套用即可。

### 9.1 策略包

**策略包概念**：
- 策略包是一组预定义的护栏规则集合，针对特定场景优化
- 用户创建团队或任务时选择适用的策略包，无需逐条配置规则
- 支持叠加使用多个策略包

**预设策略包**：

**投资与金融策略包**：
- 所有交易操作强制建议模式，不允许自动执行
- 所有结论必须附带完整证据链（来源+时间+置信度+反例）
- 单笔操作金额上限（用户可配置）
- 强制启用观察者角色作为独立风控
- 禁止在非交易时段执行交易相关操作
- 所有分析结果必须经过至少两个员工交叉验证

**软件开发策略包**：
- 代码变更必须经过Review员工审查
- 禁止直接操作生产环境
- 测试覆盖率要求（可配置）
- 敏感信息（密钥、密码）检测和拦截
- 代码提交前强制审批

**运营与内容策略包**：
- 对外发布内容必须经过审核员工和用户双重审批
- 品牌合规检查（敏感词、竞品提及等）
- 发布频率限制
- 内容原创性检查

**通用安全策略包**（默认启用）：
- Token消耗上限
- 任务执行时间上限
- 异常行为熔断
- 敏感数据脱敏

**自定义策略包**：
- 用户可以基于预设策略包修改，或从零创建
- 策略包可导出分享

**策略包版本化与回滚**：
- 每个策略包有版本号（语义化版本：主版本.次版本.补丁），每次修改自动递增
- 完整的变更记录：每次修改记录变更内容、修改人、修改时间、修改原因
- 版本对比：可以对比任意两个版本之间的差异
- 灰度发布：策略包更新时可以选择灰度发布，先应用到部分团队/任务，验证无误后再全量推送
- 快速回滚：发现策略包配置有误时，一键回滚到任意历史版本
- 回滚影响评估：回滚前展示受影响的团队和正在执行的任务列表
- 锁定机制：正在执行的任务使用的策略包版本被锁定，不受实时更新影响，任务完成后自动切换到最新版本

### 9.2 审批节点

- PM在制定执行计划时，可以在关键步骤设置人工审批节点
- 系统预设的强制审批场景：
  - 涉及外部系统写操作（如发送邮件、执行交易、修改数据）
  - 涉及资金操作
  - 任务执行计划的重大变更
- 用户也可以自定义审批规则

### 9.3 权限模型（RBAC）

系统采用"组织 → 项目 → 团队 → 任务"四层权限模型，支持企业级多人协作：

**组织层**：
- 组织管理员：管理组织成员、全局策略包、模型配置、计费
- 组织成员：在授权范围内使用平台功能

**项目层**：
- 项目负责人：创建和管理项目下的团队、分配项目预算、管理项目级策略包
- 项目成员：在项目内创建任务、使用团队

**团队层**：
- 团队创建者：配置团队成员、工具授权、协作模式
- 团队使用者：向团队发布任务

**任务层**：
- 任务创建者：创建任务、审批执行计划、查看交付物
- 任务观察者：只读查看任务进度和结果

**关键权限控制点**：
- 谁能创建/修改策略包 → 组织管理员或项目负责人
- 谁能审批涉及外部写操作的任务 → 可配置，默认为任务创建者
- 谁能启用自动模式（跳过建议模式） → 组织管理员授权
- 谁能配置触发器 → 项目负责人及以上
- 谁能查看成本数据 → 项目负责人及以上

注：单人使用场景下，用户自动拥有所有权限，RBAC对其透明无感。

### 9.4 合规与责任框架

**用户风险揭示**：
- 用户首次使用涉及决策建议类功能时，系统展示风险揭示说明
- 明确告知：AI产出的分析和建议仅供参考，不构成专业意见（投资建议、法律意见、医疗建议等）
- 用户需确认已阅读并理解风险揭示后方可继续

**免责声明管理**：
- 系统在所有决策类交付物中自动附加免责声明
- 免责声明内容可由组织管理员自定义
- 默认免责声明模板覆盖常见场景（投资建议、内容发布、数据分析等）

**地域合规策略**（远期）：
- 支持按国家/地区配置合规规则
- 某些操作在特定地区可能被禁用（如跨境数据传输限制、特定行业监管要求）
- 合规规则作为策略包的一部分，可叠加到任何场景策略包之上

### 9.5 熔断与应急机制

**自动熔断**：
- Token消耗超过预设阈值时自动暂停，通知用户
- 单个员工连续执行失败超过N次时暂停该员工，PM介入处理
- 任务总执行时间超过预设上限时暂停
- 检测到异常行为模式时暂停（如循环调用、无意义输出）

**全局应急开关（Kill Switch）**：
- 系统提供一键冻结按钮，位于全局导航栏醒目位置
- 触发后立即：
  - 暂停所有正在执行的自动模式任务
  - 禁用所有触发器（定时/Webhook均不再触发新任务）
  - 建议模式的任务不受影响（因为不产生实际操作）
- 冻结状态下：
  - 用户可以逐个审查被暂停的任务，选择恢复或终止
  - 新任务只能以建议模式创建
  - 触发器需要逐个手动重新启用
- 支持设置自动冻结条件（如：1小时内出现3次以上熔断事件时自动触发全局冻结）

**事故复盘模板**：
- 全局冻结或严重熔断事件发生后，系统自动生成事故复盘报告模板：
  - 事故时间线：从异常开始到冻结/熔断的完整事件序列
  - 影响范围：受影响的任务、团队、员工列表
  - 根因分析：系统自动分析可能的原因（策略包配置问题、员工行为异常、工具故障等）
  - 已执行操作：在异常期间实际执行了哪些外部操作（用于评估损失）
  - 改进建议：基于分析结果给出策略包调整、权限收紧等建议
- 复盘报告可由用户补充完善后归档，作为后续优化的参考

### 9.6 观察者机制

- 观察者是一个特殊角色的AI员工，不参与任务执行
- 观察者的职责：
  - 监控员工输出是否存在明显的事实错误或幻觉
  - 检测不同员工的输出之间是否存在矛盾
  - 评估任务执行是否偏离了原始目标
  - 监控整体质量和一致性
- 发现问题时：
  - 轻微问题：通知PM处理
  - 严重问题：直接暂停任务并通知用户
- 观察者是可选配置，用户可以选择是否启用

## 十一、记忆与经验沉淀系统

记忆系统是降低团队重复探索成本的核心机制。目标是让团队"越用越聪明"——同类任务执行得越多，效率越高、质量越稳定。

### 11.1 短期记忆（任务内）

- 单次任务执行过程中的上下文信息
- 包括：任务书、子任务分配、员工间传递的中间结果、PM的决策记录
- 任务结束后自动清除，不占用长期存储
- 由PM管理和分配，确保每个员工只获取相关上下文

### 11.2 员工长期记忆（跨任务）

**自动沉淀**：
- 每个员工在任务结束后，自动提取本次任务中的关键经验
- 经验类型：
  - 有效策略：哪些方法/工具/思路在本次任务中效果好
  - 踩坑记录：哪些尝试失败了，失败原因是什么
  - 数据源评价：哪些数据源质量高/低，哪些信息可靠/不可靠
  - 模式识别：发现的规律性问题或模式（如"某类财报数据通常滞后2天"）

**结构化存储**：
- 每条经验记录包含：
  - 经验摘要（一句话描述）
  - 详细内容
  - 来源任务（可追溯）
  - 适用场景标签
  - 置信度（基于验证次数，被多次验证的经验置信度更高）
  - 创建时间和最后验证时间

**自动应用**：
- 员工接到新任务时，系统自动检索与当前任务相关的历史经验
- 相关经验作为补充上下文注入员工的提示词中
- 员工可以选择采纳或忽略历史经验（但需说明理由）

**经验衰减**：
- 长期未被验证或引用的经验，置信度逐步降低
- 被多次证明无效的经验自动标记为"已过时"
- 用户可以手动审核和清理员工的经验库

### 11.3 团队记忆（跨任务）

**任务经验沉淀**：
- PM在每次任务结束后自动生成「任务复盘摘要」：
  - 任务类型和目标概述
  - 采用的协作模式和执行策略
  - 关键决策点及其选择理由
  - 哪些策略有效、哪些走了弯路
  - 最终交付质量评估
  - 资源消耗（Token、时间、成本）
- 复盘摘要按任务类型打标签，存入团队记忆库

**执行模板沉淀**：
- 当某类任务被成功执行多次后，PM可以将成熟的执行方案沉淀为「执行模板」
- 执行模板包含：推荐的子任务拆解方式、员工分配方案、关键审批节点设置
- 下次同类任务启动时，PM可以直接复用执行模板，跳过从零规划的过程
- 执行模板支持版本管理，随着经验积累持续优化

**协作经验**：
- 记录团队成员之间的协作模式：哪些员工组合配合效果好
- 记录常见的协调问题和解决方式
- 新任务组建团队时，PM参考协作经验推荐最优成员组合

**知识图谱（远期）**：
- 将团队积累的所有经验、知识、模式构建为知识图谱
- 支持跨任务、跨员工的知识关联和推理
- 例如：分析师A在任务X中发现的行业趋势 + 分析师B在任务Y中发现的政策变化 → 自动关联并提示潜在影响

### 11.4 记忆的检索与应用流程

新任务启动时，记忆系统的工作流程：

```
新任务创建
    ↓
PM接收任务目标
    ↓
系统自动检索 ──→ 团队记忆：同类任务的复盘摘要和执行模板
    │           ──→ 员工经验：各员工在相关领域的历史经验
    │           ──→ 知识库：相关领域的专业知识
    ↓
PM获得"经验包"
    ↓
基于经验包制定执行计划（复用成熟方案，跳过已知弯路）
    ↓
分配子任务时，将相关员工经验一并下发
    ↓
员工基于历史经验 + 当前上下文执行任务（减少探索性尝试）
    ↓
任务完成后，新经验自动沉淀回记忆系统
```

### 11.5 记忆管理

**用户可见可控**：
- 用户可以查看所有员工和团队的记忆内容
- 可以手动添加、编辑、删除经验条目
- 可以标记某些经验为"核心经验"（不会被自动衰减）
- 可以导出/导入记忆数据

**记忆容量管理**：
- 设置记忆条目上限，超出时自动清理低置信度、低引用的条目
- 支持按任务类型设置不同的记忆保留策略
- 提供记忆使用统计：哪些经验被引用最多、哪些从未被使用

## 十二、通知系统

### 12.1 通知场景

- 任务需要用户审批时
- 任务执行完成时
- 任务执行异常/被熔断时
- 触发器触发新任务时
- 观察者发现严重问题时

### 12.2 通知方式

- 站内通知（默认）
- 浏览器推送通知
- Webhook回调（可对接企业IM如Slack、飞书等）

## 十三、错误传播防护

多Agent系统的核心挑战是错误传播——单个Agent的错误会在传递中被放大（业界称为"17x Error Trap"），导致最终输出严重偏离预期。AgentCorp需要系统性地防护这一风险。

### 13.1 子任务输出校验

**结构化校验**：
- 每个子任务定义明确的输出格式规范（Schema）
- 员工产出结果后，系统自动校验格式是否符合规范
- 格式不合规的输出不会传递给下游，而是返回给员工要求修正

**语义校验**：
- PM在接收员工输出后，进行语义层面的质量检查：
  - 输出是否回答了子任务的问题
  - 输出是否与已知事实明显矛盾
  - 输出的置信度是否达到阈值
- 不合格的输出触发重试或人工介入

### 13.2 错误隔离

- 每个员工在独立的上下文中执行，一个员工的错误不会直接污染其他员工的输入
- PM作为信息中转站，负责过滤和清洗传递的信息
- 当某个员工的输出被判定为低质量时，PM可以：
  - 要求该员工重试（最多N次）
  - 换一个员工重新执行该子任务
  - 跳过该子任务，调整后续计划
  - 标记为需要人工介入

### 13.3 自动重试与降级

**重试策略**：
- 员工执行失败时自动重试，最多重试N次（可配置）
- 每次重试附带上次失败的原因，帮助员工调整策略
- 重试次数耗尽后升级处理

**降级策略**：
- 当首选员工持续失败时，自动切换到备选员工（如果团队中有同类角色）
- 当自动模式出现异常时，自动降级为建议模式
- 当整个子任务链路出现问题时，PM可以重新规划执行路径

### 13.4 错误传播追踪

- 当最终交付物出现问题时，可以沿着子任务依赖链路回溯，定位错误源头
- 系统记录每个子任务的输入和输出快照，支持事后审计
- 错误模式统计：哪些类型的子任务容易出错、哪些员工的错误率较高

## 十四、可观测性

可观测性是企业用户信任AI团队的基础。用户需要能够"看见"AI在做什么、为什么这么做。

### 14.1 决策摘要与行为记录

不依赖模型是否输出思考链（不同模型支持程度不一），而是通过结构化记录来实现可追溯：

- 每个员工的行为记录：
  - 接收到的输入和上下文摘要
  - 做出的决策和结论
  - 决策依据（引用了哪些数据/知识/上游输出）
  - 工具调用记录（调用了什么、为什么调用、结果是什么）
  - 输出的交付物摘要
- PM的决策记录：
  - 任务拆解方案及理由
  - 子任务分配方案及理由
  - 动态调整的触发原因和调整内容
- 所有记录由系统自动生成，不依赖模型的特定能力

### 14.2 工具调用链路追踪

- 每次工具调用记录完整信息：
  - 调用时间
  - 调用参数（脱敏后）
  - 返回结果
  - 耗时
  - 是否成功
- 支持按时间线查看完整的工具调用序列
- 异常调用高亮标记

### 14.3 Token消耗细粒度追踪

- 按员工、按子任务、按工具调用三个维度统计Token消耗
- 区分输入Token和输出Token
- 实时消耗速率监控
- 异常消耗告警（如某个员工突然消耗大量Token）

### 14.4 系统健康仪表盘

- 全局视图：当前活跃任务数、运行中的员工数、系统资源使用率
- 异常告警：熔断事件、失败任务、超时任务
- 历史趋势：任务成功率、平均成本、平均耗时的变化趋势

## 十五、成本与收益面板

### 15.1 任务前：预算预估

- 任务规划阶段，PM根据执行计划自动估算资源消耗：
  - 预估Token消耗（基于子任务数量、涉及员工数、历史同类任务数据）
  - 预估API调用成本（基于各员工使用的模型定价）
  - 预估工具调用次数和可能产生的外部费用
- 以直观的方式展示预算范围（乐观/正常/悲观三档）
- 用户可设置预算上限，超出时自动暂停

### 15.2 任务中：实时消耗

- 实时展示当前任务的资源消耗：
  - 已消耗Token数（按员工分别统计）
  - 已产生的API成本
  - 工具调用次数
  - 预算消耗进度条（已用/预算比例）
- 消耗速率异常时发出预警

### 15.3 任务后：ROI复盘

- 任务完成后生成成本复盘报告：
  - 总Token消耗和成本明细（按员工、按子任务拆分）
  - 总耗时（从创建到完成）
  - 与预估的偏差分析
  - 效率指标：每个子任务的平均成本、重试次数、一次通过率
- 历史趋势：同类任务的成本变化趋势，帮助用户优化团队配置
- 对比分析：不同团队/员工组合执行同类任务的成本对比

## 十六、团队能力评分

### 16.1 员工胜任力评分

基于历史任务表现，为每个员工自动计算胜任力评分：

**评分维度**：
- 任务完成率：分配的子任务成功完成的比例
- 输出质量：PM对其输出的采纳率、是否需要返工
- 效率：完成子任务的平均Token消耗和时间
- 稳定性：表现的一致性，是否有大幅波动
- 协作性：在团队中与其他员工配合的顺畅度（PM评估）

**评分展示**：
- 每个员工卡片上显示综合评分（如4.2/5.0）
- 可展开查看各维度的详细评分
- 评分随任务积累动态更新

### 16.2 团队效能评分

**评分维度**：
- 任务成功率：团队完成任务的比例
- 交付质量：用户对交付物的满意度（用户可评分）
- 成本效率：同类任务的成本是否在合理范围
- 协作效率：子任务间的衔接是否顺畅，返工率

### 16.3 智能推荐

- 用户创建新任务时，系统根据任务描述和历史数据，自动推荐最优团队组合
- 推荐依据：员工胜任力评分 + 任务类型匹配度 + 历史同类任务表现
- 展示推荐理由，用户可采纳或自行调整

**冷启动策略**（新团队/新员工无历史数据时）：
- 规则先验：基于员工的角色标签和系统提示词，用规则匹配推荐（如标签含"QA"的员工推荐给测试类子任务）
- 模板默认组合：场景模板自带推荐的员工组合，新用户直接使用模板推荐
- 小样本快速校准：仅需3-5次任务执行数据即可开始生成基础推荐，随数据积累持续优化
- 相似员工迁移：新员工如果是从已有员工复制而来，继承原员工的部分评分作为初始值

## 十七、核心用户流程

### 流程零：一键上手体验（首屏引导）

新用户首次进入系统时，首屏只展示三步极简流程，隐藏所有复杂配置：

**零配置试跑机制**：
- 系统内置默认托管模型（平台提供免费试用额度），新用户无需配置任何模型即可开始体验
- 场景模板自带预配置的员工和工具，用户无需手动搭建
- 试用额度用完后，引导用户配置自己的模型API Key继续使用

**Step 1 — 选场景**：
- 展示场景模板卡片（软件开发、内容创作、数据分析、运营管理等）
- 每个卡片用一句话说明适用场景，配以直观图标
- 用户点选一个场景即可

**Step 2 — 填目标**：
- 一个简洁的输入框，用户用自然语言描述想要完成的任务
- 下方提供该场景的示例目标供参考（如"帮我review这段代码的安全性"、"写一篇关于远程办公趋势的文章"）
- 无需关心团队配置、员工选择等细节

**Step 3 — 审批执行计划**：
- 系统自动基于场景模板创建团队，PM自动生成任务书和执行计划
- 用户只需审批执行计划即可启动任务
- 执行过程中可在监控面板查看进度

**渐进式展开**：
- 首次体验完成后，引导用户了解更多高级功能
- 用户可以随时从简单模式切换到完整模式，自行配置模型、工具、员工等
- 系统记住用户偏好，老用户可以直接进入完整模式

### 流程一：首次使用（完整模式）

1. 配置至少一个LLM模型
2. （可选）配置需要的工具
3. 选择一个场景模板创建团队，或从零开始创建员工和团队
4. 创建第一个任务，体验AI团队协作

### 流程二：日常使用

1. 选择团队 → 创建任务 → 与PM对话对齐需求
2. 审批任务书 → 确认团队成员 → 审批执行计划
3. 监控任务执行 → 在审批节点做决策
4. 查看交付物 → 任务归档

### 流程三：自动化使用

1. 配置触发器（定时/Webhook）
2. 设置任务模板和自动化参数
3. 系统自动触发任务，在需要审批时通知用户
4. 用户按需介入审批或查看结果

## 十九、智能测试与持续质量保障

传统软件测试依赖预定义的断言和固定的期望值，但AI系统的输出是非确定性的——同样的输入可能产生不同但都合理的输出。AgentCorp需要一套"AI测试AI"的智能测试体系，让系统能够自我验证、自我发现问题、持续改进。

### 19.1 测试体系分层

**第一层：基础设施测试（传统自动化）**
- 平台API接口的功能测试和回归测试
- 数据存储的一致性和完整性测试
- 工具调用链路的连通性测试
- 模型API的可用性和响应测试
- 前端UI的功能和兼容性测试
- 这一层使用传统的自动化测试框架即可

**第二层：Agent行为测试（AI驱动）**
- 测试单个员工在特定场景下的行为是否符合预期
- 不是检查"输出是否等于某个固定值"，而是由AI评估"输出是否合理、专业、符合角色设定"
- 测试维度：
  - 角色一致性：员工的行为是否符合其系统提示词定义的角色
  - 工具使用合理性：是否在正确的场景使用了正确的工具
  - 输出质量：分析是否有逻辑、结论是否有依据
  - 边界行为：面对模糊输入、矛盾信息、超出能力范围的请求时是否处理得当
  - 安全合规：是否遵守了策略包的约束

**第三层：团队协作测试（AI驱动）**
- 测试完整团队执行模拟任务的端到端表现
- 测试维度：
  - PM的任务拆解是否合理
  - 子任务分配是否匹配员工能力
  - 员工间信息传递是否准确
  - 异常处理是否得当（某个员工失败时PM如何应对）
  - 最终交付物的质量和完整性
  - 资源消耗是否在合理范围

### 19.2 测试场景管理

**测试用例设计**：
- 每个员工可以配置一组测试场景（Test Scenarios）
- 测试场景包含：
  - 场景描述（模拟的任务背景）
  - 输入数据（模拟的上下文和子任务）
  - 评估标准（AI评估时参考的质量维度和期望行为）
  - 预期行为描述（自然语言描述期望的行为方向，而非固定输出值）

**团队级测试用例**：
- 每个团队可以配置端到端的测试任务
- 测试任务模拟真实用户场景，从需求输入到交付物输出
- 可以基于历史成功任务自动生成测试用例（将历史任务的输入作为测试输入，历史交付物作为质量基准）

**测试用例库**：
- 系统预设常见场景的测试用例（与场景模板配套）
- 用户可以自定义和积累测试用例
- 测试用例可以随员工/团队一起导入导出

### 19.3 AI评估引擎

**评估方式**：
- 使用独立的LLM（可以是与被测员工不同的模型）作为评估者
- 评估者接收：被测员工的输入、输出、评估标准
- 评估者产出：
  - 各维度的评分（如1-5分）
  - 评分理由（可追溯）
  - 发现的问题列表
  - 改进建议

**交叉评估**：
- 对于关键测试，可以使用多个不同模型进行交叉评估
- 多个评估者的结果取共识，减少单一模型的偏见

**评估结果校准**：
- 用户可以对AI的评估结果进行人工复核
- 人工复核的结果反馈给评估引擎，持续校准评估标准
- 随着校准数据积累，评估结果越来越准确

### 19.4 持续质量监控

**变更触发测试**：
- 员工的系统提示词被修改时，自动触发该员工的行为测试
- 工具配置变更时，自动触发相关员工的工具使用测试
- 模型更换时，自动触发使用该模型的所有员工的全量测试
- 策略包更新时，自动触发相关团队的合规测试

**定期回归测试**：
- 可配置定期（如每周）自动运行全量测试
- 即使没有配置变更，模型提供商的更新也可能影响行为，定期测试可以及时发现

**生产环境质量采样**：
- 在实际任务执行中，随机抽取一定比例的子任务输出进行AI评估
- 评估结果不影响任务执行，但会记录到质量报告中
- 当采样评估发现质量下降趋势时，发出预警

### 19.5 自我改进闭环

**问题发现 → 诊断 → 建议 → 改进**：

```
测试/监控发现问题
    ↓
AI诊断引擎分析根因
    ↓
可能的根因：
  - 系统提示词不够精确 → 建议优化提示词（给出具体修改建议）
  - 工具选择不当 → 建议调整工具配置
  - 模型能力不足 → 建议更换更强的模型
  - 上下文信息不足 → 建议补充知识库或调整PM的信息分发策略
    ↓
改进建议呈现给用户
    ↓
用户审批后应用改进
    ↓
自动触发回归测试验证改进效果
```

**提示词优化助手**：
- 当测试发现员工行为偏离预期时，系统可以自动分析原因并生成优化后的系统提示词
- 优化建议以diff形式展示，用户可以选择采纳或修改
- 采纳后自动触发测试验证效果

**团队配置优化**：
- 基于测试和生产数据，系统可以建议：
  - 某个员工更适合使用另一个模型
  - 某个团队的协作模式应该调整
  - 某些子任务的拆解方式可以优化
- 所有优化建议都需要用户确认后才会生效

### 19.6 测试报告与仪表盘

**测试报告**：
- 每次测试运行生成详细报告：
  - 测试通过率（按员工、按团队、按维度）
  - 失败用例详情和AI评估理由
  - 与上次测试的对比（质量趋势）
  - 改进建议汇总

**质量仪表盘**：
- 全局质量评分趋势图
- 各员工的质量评分排行
- 质量预警列表（评分下降的员工/团队）
- 改进建议待办列表

## 二十、关键设计原则

1. **用户始终掌控**：AI团队是辅助而非替代，关键决策点必须有用户参与
2. **渐进式信任**：从需要频繁审批开始，随着用户信任度提升，可以减少审批节点
3. **透明可解释**：每个AI的决策和行为都应该可追溯、可解释
4. **安全优先**：涉及外部操作必须有护栏，宁可多确认也不要误操作
5. **简单上手，深度可定制**：场景模板让新用户快速体验，高级配置满足专业用户需求
6. **最小信息原则**：每个员工只获取完成其任务所需的最少信息，避免上下文污染
7. **可测试、可验证**：系统的每一层（基础设施、Agent行为、团队协作）都应该可测试，AI的行为质量应该可量化、可追踪、可改进
